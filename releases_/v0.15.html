Breaking Changes

Replace MethodHandle by GetObjectByClassName

Instead of passing a Func<...> to RunPatternConditionalStop(...) in PatternHandshake.Baseline() the user now passes an object of type IExecutable which contains an Execute() method.

Impact:

  • The user who writes the TestMethod creates an interface which implements IExecutable and contains properties which are used to pass parameters to Execute() and to store the results. This Interface is used in the TestMethod.
  • The user who wants to use the TestMethod has to write a Class that implements the new interface. This class should be put in the CustomCode folder.

Bug Fixes

Adding init state for pins TDI and TMS

The TDI and TMS pins have been added to the program for executing the resistance RdsOn test. These pins have a pull-up resistance according to the datasheet. After running the resistance RdsOn test, the digital part is connected but will never be set, since the pins are not included in the levels sheet. The update involves setting the TDI and TMS pins to init state off in order to avoid current leakage through the pull-up resistances.

Change the override used in BinaryTrip/Target test methods

Modify BinaryTrip/Target test methods in job CSRA/CS/VBT to use the int based version override of BinarySearch and equivalent test methods.

Impact: No impact

Improvements

Add support for additional instruments

  • UltraFLEX+: UP5000, UVS64HP
  • UltraFLEX: UP1600, HexVS, VSM, UVI80, SupportBoard

The Pins type now can resolve pins connected to these additional resources.

Impact: Enhanced support for a wider range of tester configurations.

Fix TimeDomain repeatedly queried when no TimeDomains

When pattern has no TimeDomains it will return "". TheHdw.Patterns(Name).TimeDomains

public string TimeDomain {
    get {
        if (_timeDomain is null) _timeDomain = TheHdw.Patterns(Name).TimeDomains;
        return _timeDomain;
    }
}

Impact: Reduces overhead by avoiding redundant hardware API communication.

Modify PinSite / Site <Samples> objects to utilize TestLimit() for datalogging

Swap from previous method to TestLimit() since functionality / support has been introduced for PinSite<Samples> as resultval param.

PinSite<Samples<T>>:

TheExec.Flow.TestLimit(result, CompareMode: tlLimitCompareType.EachSample, ForceVal: forceValue, ForceUnit: UnitType.Custom, CustomForceunit: forceUnit, ForceResults: tlLimitForceResults.Flow);

Site<Samples<T>>:

TheExec.Flow.TestLimit(new PinSite<Samples<T>>("", result), CompareMode: tlLimitCompareType.EachSample, ForceVal: forceValue, ForceUnit: UnitType.Custom, CustomForceunit: forceUnit, ForceResults: tlLimitForceResults.Flow);

Impact: Reduce / Remove pipeline stalling in datalogging process by using deferred method.

IG-ComponentManager integration

This PR integrates IG-ComponentManager for validating software component versions used in the test program. The implementation adds JSON-based configuration and parsing logic to verify that required software components (IG-XL, Oasis, SSL, Visual Studio, etc.) meet version requirements specified in IG-ComponentManager.json. The ComponentManager.json file will be called from IG-Link. Both IG-ComponentManager.json and IG-Link are Oasis tools and should make the versioning problem more stable and a concrete solution.

We are expecting more changes to this solution when Oasis team will fix some issues.

Details

This test demonstrates how to verify that a specific exception type is thrown when invalid input is provided. Exception‑based assertions are appropriate when failure is part of the contract of the method. Using Assert.ThrowsException<T> ensures that:

  • the correct exception type is thrown
  • the failure is explicit and intentional

This pattern is commonly used for validating defensive programming and input validation rules.

}

Details

This test verifies that a specific property assignment occurred on a dependency. Property‑based interaction tests are useful when behavior is expressed through state changes rather than method calls, such as:

  • configuration objects
  • instrument setup parameters
  • mode or flag switches

The test ensures that the unit under test correctly configures its dependency before further actions occur.

[MsTest.TestMethod]
public void Process_ForceV_PropertyWasSet() {

    _mockTheHdw.VerifySet(hdw => hdw.DCVI.Pins(pinName).Voltage.Value = voltageToSet, Times.Once); // Verify that the method was called once with the expected parameter

Process_ReadFromDependency_SetupReturnValue()

Setup: "setup return values that would have been otherwise default"

[TestMethod]
public void Process_ReadFromDependency_SetupReturnValue()

Details

This test demonstrates how to use mock setups to control the behavior of external dependencies. The hardware interaction is replaced by a mock that returns a predefined measurement result. This allows the test to:

  • execute deterministically
  • avoid real hardware access
  • focus on validating the consuming logic

Using Setup(...).Returns(...) is the standard technique for isolating a unit under test from its dependencies while preserving realistic behavior.

[MsTest.TestMethod]
public void Process_ReadFromDependency_SetupReturnValue() {
    // Arrange
    string pinName = "VDD";
    CreateFakePins([pinName], [InstrumentType.UVI264], _mockTheHdw, _mockTheExec); // Create fake pins for the test
    PinSite<double> returnValue = new(pinName, 3.3);
    _mockTheHdw.Setup(hdw => hdw.DCVI.Pins(pinName).Meter.Read(It.IsAny<tlStrobeOption>(), It.IsAny<int>(), It.IsAny<double>(), It.IsAny<tlDCVIMeterReadingFormat>()))
        .Returns(returnValue.ToPinListData());
    // Act
    PinSite<double> actualValue = TheLib.Acquire.Dc.Measure(new Pins(pinName), Measure.Voltage); // Call the method that uses the dependency
    // Assert
    MsTest.Assert.AreEqual(returnValue, actualValue); // Verify the result
}

Details

This example demonstrates the implementation of a parameterized unit test using the [DataTestMethod] and [DataRow] attributes. Parameterized tests allow the same test logic to be executed against multiple input combinations without duplicating code. Each [DataRow] defines a distinct test case that is executed independently by the test runner. This approach:

  • reduces code duplication
  • improves coverage
  • makes edge cases explicit
  • keeps related scenarios grouped together

Parameterized tests are especially useful for pure logic, validation routines, and algorithmic behavior.

[MsTest.DataTestMethod]
[MsTest.DataRow(10, 15, 25)]
[MsTest.DataRow(-5, 5, 0)]
[MsTest.DataRow(0, 0, 0)]
public void Process_ReturnsSum_Parametrized(int a, int b, int expected) {
    // Act
    int c = a + b;
    // Assert
    MsTest.Assert.AreEqual(expected, c);
}

Process_ReturnsSum_StateAssertion()

Assert: "checks certain states/objects" (Assert.AreEqual, IsTrue/False)

[TestMethod]
public void Process_ReturnsSum_StateAssertion()

Details

This example demonstrates the most fundamental form of unit testing: state verification. The test executes a small piece of logic and asserts the resulting value using basic assertion APIs such as Assert.AreEqual, Assert.IsTrue, and Assert.IsFalse.

State‑based assertions are typically used when the unit under test:

  • returns a value,
  • modifies internal state,
  • or produces a deterministic result without external side effects.

This pattern should be preferred whenever possible, as it keeps tests simple, stable, and independent of internal implementation details.

[MsTest.TestMethod]
public void Process_ReturnsSum_StateAssertion() {
    // Arrange
    int a = 5;
    int b = 7;
    // Act
    int c = a + b;
    // Assert
    MsTest.Assert.AreEqual(12, c);
    MsTest.Assert.IsTrue(c > 0);
    MsTest.Assert.IsFalse(c == 0);
}

Process_ValidInput_TimesNever()

Times.Never: a very important blueprint for "should NOT happen"

[TestMethod]
public void Process_ValidInput_TimesNever()

Details

This example demonstrates the use of Times.Never to verify that no side effects occurred. The test validates that, for valid input, the system does not raise alerts or error conditions. Negative‑behavior assertions are essential for ensuring that error paths are not accidentally triggered in normal operation. This pattern is commonly used to validate:

  • validation logic
  • safety checks
  • guard conditions
[MsTest.TestMethod]
public void Process_ValidInput_TimesNever() {
    // Arrange
    Mock<IAlertService> mockAlertService = new() { DefaultValue = DefaultValue.Mock };
    Services.Configure(mockAlertService.Object);
    // Act
    TheLib.Validate.InRange(7, 5, 10, ""); // 7 is within the range of 5 to 10
    // Assert
    mockAlertService.Verify(alert => alert.Error(It.IsAny<string>(), It.IsAny<int>(), It.IsAny<string>()), Times.Never);
}
